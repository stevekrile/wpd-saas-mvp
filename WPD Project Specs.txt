# Whole Process Design SaaS MVP — Agent Project Plan

## 1. Product Objective

Build the first usable version of a Whole Process Design SaaS application using:

* Backend: ASP.NET Core Web API
* Data store: SQL Server
* ORM: Entity Framework Core
* Frontend: React.js using Vite
* Development environment: Visual Studio
* Authentication: ASP.NET Core Identity or equivalent JWT-based authentication
* Initial deployment target: Azure App Service or equivalent cloud host, with SQL database

The MVP should create a public-facing methodology site and a low-friction free-tier diagnostic workspace. The application must be architected so later paid individual and enterprise tiers can be added without redesigning the database or authentication model.

The SaaS product should feel like an extension of the Whole Process Design methodology without exposing the full intellectual property of the book. The product should guide users into structured thinking, not reproduce the book or expose the complete internal methodology.

## 2. Core Product Strategy

### Public Tier

Purpose: Explain the methodology and create account conversion.

Capabilities:

* Public landing page
* High-level explanation of Whole Process Design
* Public overview of the four system lenses
* Examples of system distress signals
* Call-to-action to create a free account
* No access to full diagnostic logic
* No downloadable proprietary templates

### Free Member Tier

Purpose: Give users a useful “aha” moment.

Capabilities:

* Account creation
* Create one active process
* Complete a guided four-lens diagnostic
* Receive a basic diagnostic summary
* See simple lens scoring
* View a basic System Tension Map
* Save progress
* Limited export or no export
* Upgrade prompts for deeper artifact support

### Pro Individual Tier

Purpose: Support the user who wants to use WPD as a private workbench.

Planned future capabilities:

* Multiple active processes
* Full saved diagnostic history
* Artifact builders
* More detailed recommendations
* Export to PDF or Word
* Private notes
* Version history
* Personal process library

### Enterprise Tier

Purpose: Support organizational adoption.

Planned future capabilities:

* Organization workspaces
* Multiple users
* Role-based access
* Shared process library
* Admin dashboard
* SSO/SAML support
* Audit logs
* Enterprise reporting
* Configurable templates
* Enterprise license management

For MVP, build only Public and Free Member tiers, but design the schema to support Pro and Enterprise later.

## 4. Primary User Stories

### Public Visitor

As a public visitor, I want to understand what Whole Process Design is so I can decide whether to create an account.

Acceptance criteria:

* Visitor can access landing page without login.
* Visitor sees a high-level explanation of WPD.
* Visitor sees four lens names and brief descriptions.
* Visitor does not receive proprietary diagnostic logic.
* Visitor has clear path to create a free account.

### Free Member

As a free member, I want to create a process and diagnose it through the four lenses so I can identify where my system may be weak.

Acceptance criteria:

* User can register and log in.
* User can create one active process.
* User can enter process name, description, context, and problem statement.
* User can answer diagnostic questions grouped by lens.
* User can save incomplete diagnostic progress.
* User can submit diagnostic when required questions are complete.
* User receives a simple lens summary and tension map.
* User is prompted to upgrade when attempting to create a second active process or access deeper artifacts.

### Future Pro User

As a future paid individual user, I want multiple private workspaces and deeper artifacts so I can use WPD repeatedly across my work.

MVP requirement:

* Database must support subscription tier and process limits.
* Authorization should check tier capabilities through a service, not hardcoded UI-only logic.

### Future Enterprise Admin

As a future enterprise admin, I want organization-level workspaces, user roles, and shared process libraries.

MVP requirement:

* Database must include organization-ready structure even if enterprise UI is not built.
* Users should optionally belong to an organization.
* Roles should be designed with future enterprise access in mind.

## 8. API Endpoints

Use REST endpoints first. Do not overcomplicate with GraphQL.

### Public

```text
GET /api/public/lenses
GET /api/public/content/landing
```

### Auth

```text
POST /api/auth/register
POST /api/auth/login
POST /api/auth/logout
GET /api/auth/me
```

### Workspaces

```text
GET /api/workspaces
GET /api/workspaces/{id}
```

### Processes

```text
GET /api/processes
POST /api/processes
GET /api/processes/{id}
PUT /api/processes/{id}
DELETE /api/processes/{id}
```

Tier rule:

* Free users may create only one active process.
* If the limit is exceeded, return a structured error that the frontend can render as an upgrade prompt.

### Diagnostics

```text
POST /api/processes/{processId}/diagnostics
GET /api/diagnostics/{diagnosticId}
GET /api/diagnostics/{diagnosticId}/questions
PUT /api/diagnostics/{diagnosticId}/responses
POST /api/diagnostics/{diagnosticId}/submit
GET /api/diagnostics/{diagnosticId}/results
```

### Tension Map

```text
GET /api/diagnostics/{diagnosticId}/tension-map
```

### Subscription / Tier

```text
GET /api/subscription/current
GET /api/subscription/features
```

MVP note: Payment not required. These endpoints should return the current seeded tier and feature limits.

## 9. Diagnostic Scoring Logic

Build a simple scoring service in the Application layer.

### Inputs

* Diagnostic responses
* Question weights
* Lens grouping

### Outputs

* Score per lens
* Normalized score per lens
* Score band per lens
* Primary tension lens
* Basic summary
* Basic tension map records

### Scoring Rules for MVP

Use simple numeric questions rated 1–5.

Example scoring concept:

* 1 = Strong disagreement / high weakness
* 3 = Mixed or unclear
* 5 = Strong agreement / low weakness

Invert the scoring into a “tension score” if needed:

* Higher tension score = more likely weakness
* Lower tension score = less likely weakness

Keep the scoring logic in one service:

```text
IDiagnosticScoringService
```

Do not scatter scoring logic across controllers or React components.

## 17. Recommended Technical Decisions

### Backend

* Use ASP.NET Core Web API.
* Use Entity Framework Core.
* Use SQL Server.
* Use DTOs rather than exposing EF entities directly.
* Use FluentValidation or equivalent for request validation.
* Use Swagger/OpenAPI.
* Use centralized error handling middleware.
* Use dependency injection for all services.

### Frontend

* Use React with Vite.
* Use TypeScript.
* Use React Router.
* Use a query library such as TanStack Query or a simple API service layer.
* Use component-level form validation.
* Keep business scoring logic out of the frontend.
* Use environment variables for API URL.

### Database

* Use EF Core migrations.
* Seed reference data.
* Use GUID or integer IDs consistently.
* Include CreatedAt and UpdatedAt fields on major entities.
* Avoid premature multi-tenant complexity, but include Organization and Workspace from the beginning.

---

## 18. Nonfunctional Requirements

### Security

* Passwords must be managed through Identity or a secure authentication provider.
* Do not store secrets in source control.
* Protect all app API routes.
* Validate user ownership before returning process or diagnostic data.
* Enforce tier limits server-side.

### Performance

* MVP does not need advanced caching.
* Use pagination for future list endpoints.
* Avoid loading all diagnostic history unnecessarily.

### Maintainability

* Keep domain logic out of controllers.
* Keep scoring logic in a dedicated service.
* Use migrations for schema changes.
* Keep frontend API calls centralized.
* Use clear naming consistent with WPD concepts.

### Privacy

* Treat process descriptions as potentially sensitive business information.
* Do not use user-entered process data for public examples.
* Add placeholder privacy language on the site until formal legal text is drafted.

## 21. Critical Guardrails for the Agent

Do not build beyond MVP unless explicitly requested.

Do not expose proprietary scoring rules in React.

Do not turn the site into a book-content repository.

Do not build enterprise features yet.

Do not add payment processing yet.

Do not add AI features yet.

Do not create a complex process mapping canvas yet.

Prioritize a clean working vertical slice over feature breadth.

The first successful product experience is this:

A user creates an account, names one messy process, answers guided questions, and sees a useful four-lens diagnosis that makes them feel clearer about what is wrong.
